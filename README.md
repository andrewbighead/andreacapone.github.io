# Andrea Capone — Personal Website

Static site hosted on GitHub Pages. Content is data-driven: publications,
conferences, projects, experiences, and **per-paper pages** are all generated
from JSON files in `data/`.

## Structure

```
.
├── index.html                  # Static shell (header, hero, skills, contacts)
├── styles.css                  # Visual styles (unchanged)
├── main.js                     # Renders dynamic sections on the home page
├── build_papers.py             # Generates per-paper HTML from publications.json
├── pic.jpg                     # Profile photo
├── cv.pdf                      # Your CV
├── data/
│   ├── publications.json       # Source of truth for papers + theses
│   ├── conferences.json
│   ├── projects.json
│   └── experiences.json
├── papers/                     # GENERATED by build_papers.py (do not hand-edit)
│   ├── iatl/
│   │   ├── iatl.html           # ← generated
│   │   └── IATL_KR_Camera_Ready.pdf
│   ├── ictl/…
│   └── landslide_forecasting/…
└── .github/workflows/build-papers.yml   # Auto-regenerates on push
```

## How content flows

```
data/publications.json
         │
         ├─→ main.js           → rendered on index.html (home)
         │
         └─→ build_papers.py   → generates papers/<slug>/<slug>.html
                                  (static pages with Scholar meta tags)
```

Everything is driven by **one file**: `data/publications.json`.

## Adding a new paper

1. Drop the PDF in a new folder: `papers/<slug>/<paper>.pdf`
2. Add an entry in `data/publications.json` under `"papers"` (newest first):

```json
{
  "id": "my-new-paper-2026",
  "slug": "my_new_paper",
  "year": "2026",
  "month": "03",
  "title": "Full paper title",
  "authors": ["First Author", "Andrea Capone", "…"],
  "venueShort": "VENUE 2026",
  "venueFull": "Main Track Conference Paper, VENUE 2026",
  "conferenceTitle": "Proceedings of VENUE 2026",
  "type": "conference",
  "badge": "VENUE 2026 • Main Track",
  "location": "City, Country",
  "pages": "10–20",
  "firstPage": "10",
  "lastPage": "20",
  "publicationDate": "March 2026",
  "doi": "10.xxxx/xxxxx",
  "officialUrl": "https://doi.org/10.xxxx/xxxxx",
  "officialLabel": "Official page",
  "doiLabel": "DOI",
  "doiText": "https://doi.org/10.xxxx/xxxxx",
  "doiUrl": "https://doi.org/10.xxxx/xxxxx",
  "abstract": "Full abstract…",
  "bibtex": "@inproceedings{…}",
  "pageUrl": "papers/my_new_paper/my_new_paper.html",
  "pdfUrl": "papers/my_new_paper/my_new_paper.pdf",
  "pdfFilename": "my_new_paper.pdf"
}
```

3. Regenerate the paper page:

```bash
python3 build_papers.py
```

4. Commit and push. If you use the provided GitHub Action, you can skip step 3
   — the workflow regenerates paper pages on every push to `main` and commits
   them back.

### Required fields for Google Scholar indexing

For a paper page to be indexed by Scholar, the generator needs at least:
`title`, `authors` (array), `year`, `pdfFilename`, and `pageUrl`.
`build_papers.py` will refuse to run and print an error if any of these are
missing for a paper that has a `pageUrl`.

Strongly recommended for quality indexing: `conferenceTitle` **or**
`journalTitle`, `month`, `firstPage`/`lastPage`, and `doi`.

### Optional fields

| Field          | Shown where                                        |
| -------------- | -------------------------------------------------- |
| `badge`        | Colored tag at the top of the paper page           |
| `location`     | Meta line on paper page                            |
| `officialUrl`  | Second button next to "Download PDF"               |
| `doiUrl`       | Clickable link in the DOI line on the home page    |

## Validating and regenerating

```bash
# Regenerate paper pages (idempotent — only writes files that changed)
python3 build_papers.py

# Dry-run: report what would change, exit 1 if anything is stale
python3 build_papers.py --check

# Quiet mode
python3 build_papers.py --quiet
```

## CI: the GitHub Action

`.github/workflows/build-papers.yml` does two things:

- **On pull requests** that touch `publications.json` or the generator: runs
  `python3 build_papers.py --check` and fails the PR if the paper HTML files
  are not up to date. This prevents merging a JSON change without the
  corresponding HTML update.
- **On push to `main`**: regenerates the pages and commits the result back to
  the repo with message `chore: regenerate paper pages [skip ci]`. This means
  you can edit `publications.json` directly on GitHub (web UI) and the site
  updates itself.

If you don't want the auto-commit behaviour, delete the last step of the
workflow; you'll then need to run `build_papers.py` locally before pushing.

## Local preview

`fetch()` won't load JSON from `file://` — you need a local server:

```bash
python3 -m http.server 8000
# then open http://localhost:8000
```

## What's in a generated paper page?

Each generated page contains:

- `<title>`, `<meta name="description">`, `<link rel="canonical">`
- Full set of Google Scholar `citation_*` meta tags:
  `citation_title`, `citation_author` (one per author), `citation_conference_title`
  or `citation_journal_title`, `citation_publication_date`, `citation_firstpage`,
  `citation_lastpage`, `citation_doi`, `citation_pdf_url`, `citation_abstract_html_url`,
  `citation_language`
- Open Graph + Twitter Card tags for link previews on social platforms
- The same visual layout as before (Back-to-home link, badge, title, authors,
  meta line, PDF + Official-page buttons, abstract, BibTeX)
- Font Awesome stylesheet (was missing in the old hand-written pages, which
  is why the PDF icon wasn't showing)
